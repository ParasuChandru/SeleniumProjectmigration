Test Automation Strategy 2025-2026
1. Purpose
The purpose of this document is to ensure that test automation in LocalTapiola is implemented
effectively and efficiently within a software development lifecycle. It provides clear plan and
framework to guide teams in automating repetitive testing, improving the speed and the
accuracy of testing, and thus supporting continuous integration and delivery.

This document presents version 1.0 of the Test Automation Strategy for the years 2025-

High-level Vision
The standardization of test automation at LocalTapiola is set to enhance both the speed
and reliability of our testing processes, ultimately elevating the quality of the systems
developed for our customers. By implementing a unified testing framework and
infrastructure, we are positioning ourselves to support release on-demand model and
achieving true shift-left practices, paving the way for cost-effective test automation in the
future.
The success of test automation development hinges on excellent collaboration between
LocalTapiola and our vendors. This partnership is crucial for fostering a cultural shift that
positions test automation as a cornerstone of our quality assurance practices.
Furthermore, we are actively exploring the potential of artificial intelligence to supercharge
our test automation efforts, ensuring that we remain on the right path moving forward.

5. Test Environments
Test automation should be thoughtfully designed and implemented to enhance the usability and
scalability of test suites and cases across different environments. We should be able to use the
same test automation test suites and cases in every possible environment with slight changes.
To achieve this, it is essential to store environment variables/urls/paths in variables rather than
hard-coding them directly into test cases.

To ensure optimal performance, each test case should be designed to run independently. This

on previous cases.

equipped with the knowledge to resolve any specific test case

Test automation should, by default, generate the necessary test data used. This approach aims
to ensure that all test data at LocalTapiola is synthetic in the future. However in some cases
generation of test data is not possible for more complex cases. These situations has to be

Both LocalTapiola and its vendors should work collaboratively towards establishing standardized
methods for test data creation.

8. Test Automation Tools
Robot Framework (Primary test automation framework)
Robot Framework is set to become the primary test automation framework for the foreseeable
future.
Currently, many system-level test automations are built using the Java + Selenium test
automation framework. It is essential to explore the most effective methods for maintaining Java
+ Selenium at an optimal level while also investigating the best approaches to transition these
tests to Robot Framework. However, it is important to note that in some instances, such a
conversion may not be feasible.
Java + Selenium testing frameworks (Secondary test automation framework)
Java and Selenium testing frameworks are primarily employed by TCS and Toivo projects.
LocalTapiola continues to utilize Java and Selenium in ongoing projects, as well as for the
regression sets developed in the past, as long as necessary.
LocalTapiola is set to begin planning for potential migrations from Java + Selenium to the Robot
Framework in the Spring of 2025. The initial projects under consideration for this migration
include Aito Digi, other Digi projects, and Toivo. At this stage, we lack clear visibility on whether
all projects can be migrated and the specific timelines for each transition.
It is recommended that no new test automation projects utilizing Java and Selenium be initiated.
Browserstack
BrowserStack offers a comprehensive solution for testing across a diverse range of browsers
and mobile devices.


While the use of BrowserStack is highly encouraged, it is not a strict requirement. LocalTapiola
is committed to developing templates for the Robot Framework, which facilitate seamless
integration with BrowserStack whenever necessary.
Azure DevOps pipelines
LocalTapiola has made substantial investments in migrating to a cloud platform. Currently, many
of our regression test automations are executed on local servers. Our objective is to transition
from these servers to utilizing Azure DevOps environments.
This shift will significantly reduce the maintenance required for various environments, such as
Jenkins.
Transitioning to Azure DevOps pipelines presents considerable advantages compared to other
tools. By moving away from Jenkins pipelines, we can not only improve the scalability of our
DevOps practices across different projects but also lower long-term infrastructure maintenance
costs.
Azure DevOps pipelines enable DevOps teams at LocalTapiola to implement a release-on-
demand model. As a result, migrations will be pivotal in this transition.
Azure Keyvault
Utilizing Azure Key Vault as the primary storage solution for various test automation-related
secrets and passwords is highly recommended. Azure Key Vault can be seamlessly integrated
into Azure DevOps pipelines, allowing for the retrieval of secrets without the need for managing
additional secret files. This approach simplifies the process, especially when passwords and
secrets require updates.
Utilizing tools such as Keepass raises significant security concerns, primarily due to the
challenges in managing access to passwords, sensitive information, and user rights.
LocalTapiola strongly advocates for transitioning away from tools like Keepass as swiftly as
possible.
9. Documentation
Test automation must be documented properly, adhering to established best practices. It should
be regarded as a form of software development rather than merely a collection of scripts or
gimmicks. Proper documentation will significantly ease maintenance efforts. This chapter
outlines key principles to follow in LocalTapiola's test automation, which can certainly be
expanded upon.

b. Test automation maturity metric
Introducing a new metric for assessing test automation maturity. Tracking these KPIs helps
LocalTapiola to understand our test automation maturity, identify areas for improvement and
demonstrate the value of automation over time. This will also help us to keep track of future
improvements made in the field of test automation at LocalTapiola.
Test Automation Maturity KPI’s
SQ1-Strategy
We have a test automation strategy that defines ‘what test scope will be automated to what
degree, when, by whom, by which methods, by what test tools, in what kind of
environment’.
SQ2-Resources
We allocate enough resources for test automation, e.g., skilled people, the funding, the time
& effort, test environment with the required software, hardware, or test data for test
automation.
Why: To gain insights of test automation maturity at LocalTapiola and to assess its
current state.
When: Twice a year, at the beginning of each half.
Dates:
2025 H1 
1st Question form will be sent on 27.01.2025 (time to answer 1 week)
Going through the results during Q1, and making improvements to strategy
Results Presentation: Week 9
2nd Question form will be sent on 01.08.2025 (time to answer 2 weeks)
Going through the results during Q3, and making improvements to strategy
Results Presentation: Week 37-38
How: Inquiries targeting test automation developers and test managers will be
conducted using a form based on test automation maturity KPIs. The completed forms
will be sent via email, and the results will be thoroughly evaluated.
By who: The head of test automation at LocalTapiola will initiate inquiries, and the
results will be evaluated in partnership with the Quality Assurance managers.
Purpose: 
Elevating Test Automation Practices to New Heights: Gaining Strategic Insights into
Existing Gaps and Areas for Improvement


SQ3-Roles
We clearly define roles and responsibilities of stakeholders in test automation.
SQ4-Knowledge 
We are systematically learning from prior projects. We collect and share expertise, good
test automation practices, and good test tools for future projects.
SQ5-Competence
Our test team has enough expertise and technical skills to build test automation based on
our requirements.
SQ6-Tools
We currently have the right test tools that best suit our needs.
SQ7-Test environment 
We have control over the configuration of our test environment.
SQ8-Guidelines
We have guidelines on designing and executing automated tests. Those guidelines include,
e.g., coding standards, test-data handling methods, specific test design techniques to
create test cases, processes for reporting and storing test results, the general rules for test
tool usage, or information on how to access external resources.
SQ9-Prioritization 
We effectively prioritize and schedule automated tests for the execution.
SQ10-Results
We are capable to manage and integrate test results collected from different sources (e.g.,
different test tools, test levels, test phases) into a big picture, and then report useful
information to the relevant stakeholders.
SQ11-Process
We organize our test automation activities in the stable and controllable test process.
SQ12-SUT 
Our Software Under Test enables us to conduct our test automation, e.g., maturity, running
speed, or testability of our Software Under Test is not a problem for our test automation.
11. Training
From the perspective of a test automation developer, it is essential to offer training on the new
libraries, tools, and methodologies developed for all stakeholders involved in test automation at
LocalTapiola. This initiative will facilitate knowledge sharing and help dismantle existing silos
within the team.


We are establishing a professional development pathway for manual testers at LocalTapiola.
This initiative will enable our manual testers to cultivate a culture of test automation within the
LocalTapiola Quality Assurance sector. Gaining even a minimal understanding of test automation
practices can significantly enhance manual testers' comprehension of our automation efforts
and objectives. This knowledge not only clarifies our processes but also provides valuable
insights that can improve our testing practices.
The Head of Test Automation at LocalTapiola, along with the LocalTapiola Test Automation
Team, will take charge of collecting ideas and resources for upcoming training sessions. We are
always open to hearing ideas from our vendors regarding this matter and are eager to
strengthen our culture through the training they provide.
12. Continuous Improvement
Continuous improvement in test automation necessitates a proactive strategy for identifying
gaps, embracing innovation, and empowering teams. By cultivating a culture of enhancement
and systematically tackling inefficiencies, LocalTapiola can transform our test automation into an
invaluable asset for delivering high-quality software more efficiently.
